# 商家对账未匹配维护与净额比例设计

## 背景

当前商家对账的明细匹配逻辑以动账商品编码为入口，先匹配红单采购，再通过采购中的商家、直播日期和直播间匹配红单货款。现有实现存在两个业务问题：

- 动账文件中的商品编码不能再作为完整计算依据。系统应只从动账明细读取 `product_code`，其余大商家、直播间、直播日期和收款商家信息都应来自红单采购和红单货款。
- 匹配失败后缺少面向财务人工维护的入口。财务需要导出未匹配数据和去重货号汇总，维护红单后重新上传并复核。

另外，当前 `应付商家净额` 相关计算写死为 `GMV * 70%`。后续每个月的比例可能不同，需要提供维护入口，并按月度配置计算。

## 目标

- 新增商家对账未匹配维护入口，默认查询未匹配动账明细。
- 新增未匹配维护导出，包含未匹配明细 sheet 和按货号去重的汇总 sheet。
- 调整红单匹配口径：动账只提供商品编码，采购和货款字段均来自红单。
- 增加 `应付商家净额` 百分比维护能力，按组织、业务年月、平台保存一份比例。
- 所有明细、汇总、导出、银行差异计算使用同一套月度净额比例。
- 保持现有红单重新上传机制：同期间旧红单软删除，只使用最新版红单重新计算匹配结果。

## 非目标

- 本期不支持按店铺维护净额比例，但数据模型预留可空 `shop_id` 以便后续扩展。
- 本期不新增人工匹配结果表，也不把未匹配修正直接写回动账明细。
- 本期不改变红单上传文件结构。
- 本期不把商家名称、直播间或直播日期做模糊匹配。能匹配就精确命中，不能匹配就进入未匹配维护。

## 匹配口径

### 数据来源

动账明细表 `fin_douyin_dongzhang_details` 只使用：

- `product_code`
- 原始展示字段，例如动账流水号、商品名称、GMV、订单号和店铺信息

红单采购表 `fin_merchant_red_sheet_purchases` 提供：

- 大商家名称：`merchant`
- 直播间：`live_room`
- 直播日期：`live_date`
- 直播编号：`live_code`
- 新直播编码：`normalized_live_code`

红单货款表 `fin_merchant_red_sheet_payments` 提供：

- 收款商家：`receipt_merchant`
- 结算主体：`settlement_subject`
- 收款主体：`receipt_subject`
- 业务费用、冲减金额、应付货款金额和已付金额

### 匹配流程

1. 从动账明细读取 `product_code`。
2. 如果 `product_code` 为空，返回未匹配：`动账商品编码为空`。
3. 读取同组织、同平台、同业务年月、同店铺范围内的最新红单批次。
4. 用 `product_code` 匹配采购表的 `normalized_live_code` 和 `live_code`。
5. 采购候选行按业务唯一键归并：同组织、同年月、同商家、同直播间、同直播日期、同直播编号。
6. 每个业务唯一键取最新一条采购明细。最新优先级为红单批次创建时间、红单批次 ID、采购明细 ID。
7. 用采购行的 `merchant + live_room + live_date` 匹配同红单上下文内的货款表。
8. 货款命中且 `receipt_merchant` 非空时，明细匹配成功。
9. 货款未命中或收款商家为空时，明细保持未匹配。

### 匹配失败原因

匹配失败是业务状态，不是系统异常。返回和导出中应明确展示原因：

- `动账商品编码为空`
- `未匹配到红单采购`
- `匹配到红单采购但未匹配到红单货款`
- `匹配到红单货款但收款商家为空`

如果同一个商品编码命中多个采购业务唯一键，系统不直接报错。只要其中一条采购可以继续命中货款并获得收款商家，就使用可闭环的采购和货款结果；如果都不能闭环，则返回未匹配，并在导出中带出已识别到的采购字段，方便人工维护。

## 未匹配维护入口

### 前端入口

在商家对账模块新增页面：`未匹配维护`。

页面默认查询 `match_status=unmatched` 的动账明细，定位为人工维护工作台，而不是普通明细页的一个筛选条件。

### 筛选条件

- 业务年月，必选。
- 组织，按现有用户权限展示。
- 店铺，沿用商家对账现有店铺选择能力。
- 商品编码或货号关键词。
- 匹配失败原因。

### 列表字段

- 组织
- 店铺
- 业务年月
- 动账时间
- 动账流水号
- 商品编码
- 商品名称
- GMV
- 已识别大商家
- 已识别直播间
- 已识别直播日期
- 收款商家
- 匹配失败原因

页面操作：

- 查询
- 导出未匹配维护表
- 跳转红单上传
- 重传红单后重新查询

### 后端接口

新增专用接口，避免前端依赖通用明细页参数拼装：

- `GET /merchant-reconciliation/unmatched-details`
- `GET /merchant-reconciliation/unmatched-details/export`

查询接口返回分页数据和统计信息。导出接口返回 Excel，默认导出当前筛选条件下的全部未匹配数据。

接口内部复用商家对账匹配器，不能重新实现一套匹配逻辑。

## 未匹配导出

导出文件包含两个 sheet。

### 未匹配动账明细

保留每条未匹配动账行，字段包括：

- 组织
- 店铺
- 业务年月
- 动账时间
- 动账流水号
- 商品编码
- 商品名称
- 订单号
- 子订单号
- GMV
- 已识别大商家
- 已识别直播间
- 已识别直播日期
- 收款商家
- 匹配失败原因

### 货号汇总

按商品编码去重，用于人工维护红单。字段包括：

- 商品编码
- 商品名称示例
- 未匹配行数
- GMV 汇总
- 涉及组织
- 涉及店铺
- 已识别大商家
- 已识别直播间
- 已识别直播日期
- 失败原因汇总

商品编码为空的行汇总到空编码组，失败原因应保留 `动账商品编码为空`。

## 净额比例维护

### 维护入口

在商家对账模块新增页面或抽屉：`净额比例维护`。

入口位置建议放在商家对账汇总页工具栏，靠近现有期初余额维护。两个入口性质类似，都是影响汇总计算的月度人工维护项。

### 维护维度

本期按以下维度保存一份比例：

- 组织
- 业务年月
- 平台

默认平台为 `douyin`。未维护时默认比例为 `70%`。

### 数据模型

新增表：`fin_merchant_net_rate_settings`。

字段：

- `id`：主键。
- `org_id`：组织 ID。
- `platform_code`：平台编码，默认 `douyin`。
- `shop_id`：店铺 ID，可空，本期不启用，预留后续店铺级比例。
- `accounting_year`：业务年份。
- `accounting_month`：业务月份。
- `accounting_period`：业务年月，格式 `YYYYMM`。
- `net_rate`：净额比例，建议用 `NUMERIC(8, 6)` 存储小数，例如 `0.700000`。
- `remark`：备注。
- `created_by`、`updated_by`：维护人。
- `created_at`、`updated_at`：时间。
- `is_deleted`：软删除标记。

唯一约束：

`org_id + platform_code + accounting_period + shop_id`，仅对未删除记录生效。由于 `shop_id` 可空，迁移中需要用表达式索引或约定空店铺统一写入 `0`，避免 PostgreSQL 的 NULL 唯一约束语义导致重复。

### API

新增接口：

- `GET /merchant-reconciliation/net-rate-settings`
- `POST /merchant-reconciliation/net-rate-settings`

查询接口按组织、平台和年月返回当前比例。没有记录时返回默认 `70%`，并标记为未维护。

保存接口支持批量 upsert，便于总公司管理员一次维护多个可见组织的当月比例。

权限沿用商家对账组织可见范围：

- 超级管理员可维护所有组织。
- 总公司组织管理员可维护自己组织和直属商家组织。
- 商家管理员只可维护自己组织。

### 计算影响

所有原 `GMV * 70%` 的位置改为读取月度比例：

- 明细匹配结果中的 `live_amount`。
- 明细统计中的 `total_live_amount`。
- 汇总中的 `merchant_payable_net_amount`。
- 红单费用分摊、银行流水分摊权重。
- 应付货款余额、未付流水和银行付款差。
- 明细导出、汇总导出和附带明细导出。

如果某个组织、月份、平台没有维护比例，统一使用默认 `0.7`。默认值必须在服务层集中定义，避免不同模块各自写死。

### 页面交互

净额比例维护页面或抽屉展示：

- 组织
- 平台
- 业务年月
- 当前比例
- 是否已维护
- 备注
- 更新时间

输入控件使用百分比形式，例如输入 `70` 表示 `70%`，后端保存为 `0.700000`。前端需要限制合理范围，建议允许 `0` 到 `100`，后端同样校验。

保存后刷新汇总页，让页面使用新比例重新计算结果。

## 数据流

### 未匹配维护流程

1. 财务进入 `未匹配维护`。
2. 选择业务年月和店铺，查询未匹配动账。
3. 系统按最新红单批次执行匹配链路，返回未匹配行和失败原因。
4. 财务导出维护表。
5. 财务根据 `货号汇总` 维护红单采购或货款。
6. 财务重新上传红单。
7. 系统软删除同期间旧红单批次，保存新红单。
8. 财务回到 `未匹配维护` 重新查询，确认未匹配减少或清零。

### 净额比例计算流程

1. 页面请求明细、汇总或导出。
2. 服务层根据组织、业务年月和平台批量加载净额比例。
3. 没有维护记录的组织使用默认比例 `0.7`。
4. 匹配器使用比例计算每行动账 `live_amount`。
5. 汇总构建器累计 `live_amount` 得到 `应付商家净额`。
6. 后续应付货款余额、未付流水和银行差异沿用现有公式，但输入金额已经是新比例口径。

## 错误处理

- 未选择业务年月时，前端提示用户选择业务年月。
- 净额比例小于 0 或大于 100% 时，后端返回业务错误。
- 无权限维护其他组织比例时，后端返回业务错误。
- 红单采购或货款缺失不抛系统异常，只返回未匹配状态。
- 导出没有未匹配行时仍生成带表头的 Excel，方便用户确认当前筛选条件下无待维护数据。

## 测试策略

### 后端测试

新增或调整测试覆盖：

- 动账只使用 `product_code`，商品名称不再兜底提取编码。
- 商品编码匹配红单采购的 `live_code` 和 `normalized_live_code`。
- 采购命中后通过 `merchant + live_room + live_date` 匹配红单货款。
- 货款缺失、收款商家为空、商品编码为空时返回正确失败原因。
- 同一商品编码多条采购时，优先选择能命中货款并获得收款商家的结果。
- 未匹配维护查询默认只返回未匹配数据。
- 未匹配导出包含 `未匹配动账明细` 和 `货号汇总` 两个 sheet。
- 未维护净额比例时默认 70%。
- 维护 65% 后，明细 `live_amount`、汇总 `应付商家净额`、银行分摊权重和导出金额同步变化。
- 不同组织同月可维护不同净额比例。
- 用户不能维护不可见组织的净额比例。

### 前端测试

新增或调整测试覆盖：

- 未匹配维护页面默认带未匹配筛选。
- 导出按钮传递正确筛选条件。
- 净额比例输入按百分比展示和提交。
- 保存比例后刷新汇总数据。

## 实施顺序

1. 新增净额比例设置表和后端 schema。
2. 新增净额比例查询、保存服务和 API。
3. 改造匹配器和汇总构建逻辑，统一读取月度净额比例。
4. 改造红单采购到货款的匹配口径和失败原因。
5. 新增未匹配维护查询和导出服务。
6. 新增前端未匹配维护入口。
7. 新增前端净额比例维护入口。
8. 补充后端和前端测试。

## 验收标准

- 未匹配维护入口可以按业务年月查询未匹配动账。
- 未匹配导出包含明细和去重货号汇总。
- 动账明细只使用 `product_code` 进入红单匹配。
- 大商家、直播间、直播日期和收款商家均来自红单采购和货款。
- 重传红单后，未匹配维护查询能反映最新红单结果。
- `应付商家净额` 不再固定 70%，而是按组织、业务年月、平台配置计算。
- 未维护净额比例时仍按 70% 计算，保持历史默认口径。
- 明细页、汇总页、汇总导出、明细导出和附带明细导出口径一致。
