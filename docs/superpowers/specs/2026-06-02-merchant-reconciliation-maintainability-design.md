# 商家对账可维护性优化设计

## 背景

当前商家对账功能集中在 `backend/app/services/merchant_reconciliation_service.py`。该文件同时承担红单模板、红单导入、银行流水导入、动账明细匹配、费用分摊、汇总构建、导出和审计日志等职责。

现有实现中有多处 `JOIN`，但这些 `JOIN` 的性质不同：

- 明细列表中的组织名、店铺颜色 `outerjoin` 主要是展示字段补充，风险较低。
- 红单、动账明细、银行流水之间的匹配不是单纯 SQL join，而是业务规则匹配。
- 汇总页会在每次请求时动态推导匹配结果、分摊金额、银行流水金额和状态，可读性和定位成本较高。

本阶段优化目标优先级是可维护性。第一阶段不改变数据库模型、不改变 API 返回结构、不要求消灭所有 `JOIN`，而是拆清服务边界，让每类查询和业务规则有明确归属。

## 目标

- 保持现有商家对账业务口径和 API 行为不变。
- 将导入、查询、匹配、汇总、银行流水、导出拆成边界清晰的模块。
- 将展示型 `JOIN` 和业务匹配规则分离，避免业务层直接散落 SQL 细节。
- 为后续结果快照表或预聚合优化预留接口。
- 在关键业务规则处补充中文注释，帮助后续维护者理解匹配口径。

## 非目标

- 不在第一阶段新增商家对账快照表。
- 不把所有 Python 匹配逻辑强行改成 SQL CTE。
- 不改变现有前端页面、接口路径、字段名和错误响应格式。
- 不引入新的主体别名、日期标准化配置或人工匹配功能。

## 推荐方案

采用“服务边界重构，不改数据模型”的渐进式方案。

相比把所有逻辑改写成 SQL CTE，该方案更符合可维护性优先目标。相比直接新增结果快照表，该方案风险更低，也能先沉淀稳定接口，后续如果要落库快照，可以复用这些模块输出。

## 模块边界

### MerchantRedSheetImporter

负责红单相关导入逻辑：

- 构建红单模板。
- 解析货款 sheet 和采购 sheet。
- 校验 sheet 名、月份、结算状态和店铺。
- 根据同组织、同平台、同账期、同店铺规则软删除旧批次。
- 写入红单批次、采购明细和货款明细。

需要保留中文注释说明：货款 sheet 没有独立店铺列，系统通过采购 sheet 的商家、直播日期、直播间组合键反推出店铺。

### MerchantBankFlowImporter

负责银行流水导入逻辑：

- 解析文件名账期。
- 识别企业网银和中国邮政格式。
- 过滤直播相关流水。
- 解析账户名称、对方户名、交易时间、金额、直播日期和流水号。
- 根据同组织、同账期、同账户名称规则软删除旧批次。
- 写入银行流水文件和银行流水明细。

需要保留中文注释说明：银行流水只纳入 `flow_amount > 0` 的实际付款流出参与汇总，收入和冲正类流水只保留在明细。

### MerchantReconciliationQueries

集中承载商家对账 SQL 查询：

- 红单批次列表。
- 红单货款明细列表。
- 红单采购明细列表。
- 银行流水文件和流水明细列表。
- 动账明细基础分页、总数、组织名和店铺颜色补充。
- 财务汇总 BIC、运费险聚合。
- 期初余额读取。

该模块允许存在展示型 `outerjoin`。业务模块应调用语义化查询方法，而不是直接拼接 SQL。

### MerchantReconciliationMatcher

负责动账明细和红单之间的业务匹配：

- 从动账商品编码或商品名称提取商品编码。
- 读取同组织、同店铺、同账期的最新红单上下文。
- 用商品编码匹配红单采购。
- 用采购中的商家、直播日期、直播间匹配红单货款。
- 计算匹配状态和匹配失败原因。
- 计算 BIC、运费险和直播款分摊字段。

匹配失败不是系统异常，而是业务结果，应返回 `match_status=unmatched` 和明确的 `match_error`。

### MerchantBankFlowMatcher

负责银行流水进入汇总时的匹配和分摊：

- 读取同组织、同账期且已解析直播日期的付款流出流水。
- 优先通过红单付款桥接银行流水。
- 如果付款行已关联多个汇总组，则按汇总组 `live_amount` 权重分摊银行流水金额。
- 未命中红单付款时保留当前主体名称兜底匹配逻辑。

需要保留中文注释说明：银行流水优先桥接红单付款，再按汇总权重分摊；兜底主体匹配只是兼容路径，不是主匹配口径。

### MerchantReconciliationSummaryBuilder

负责汇总行构建：

- 从匹配后的动账明细聚合 GMV、直播款和行数。
- 合并红单货款中的业务费用、其他冲减、应付货款和已付金额。
- 合并期初余额。
- 合并银行流水金额。
- 刷新应付货款余额、未付流水、银行付款差和银行状态。
- 支持关键字和银行状态过滤。

该模块只处理业务聚合，不直接关心 API、审计日志或上传文件解析。

### MerchantReconciliationExporter

负责导出：

- 汇总导出复用 `MerchantReconciliationSummaryBuilder` 结果。
- 明细导出复用 `MerchantReconciliationMatcher` 结果。
- 导出列定义保持不变。

导出不得重新实现一套独立计算逻辑，避免页面和导出口径漂移。

### MerchantReconciliationService

保留为 API 门面：

- 校验参数。
- 解析用户权限。
- 调用导入器、查询模块、匹配器、汇总构建器和导出器。
- 记录审计日志。
- 保持原有公共方法签名，降低 API 层改动。

## 数据流

### 红单导入

API 调用门面服务。门面校验月份、权限和文件格式后交给 `MerchantRedSheetImporter`。导入器解析采购和货款 sheet，解析店铺，软删除同期间旧批次，写入红单批次、采购明细和货款明细。

### 银行流水导入

API 调用门面服务。门面处理上传文件和权限，`MerchantBankFlowImporter` 解析文件名账期、读取 Excel、过滤直播相关流水、解析标准字段并写入批次和明细。

### 明细页

`MerchantReconciliationQueries` 读取动账明细分页、总数、组织名和店铺颜色。`MerchantReconciliationMatcher` 批量加载同组织、同店铺、同账期的红单上下文，输出商品编码、商家主体、收款商家、直播日期、匹配状态、分摊 BIC 和运费险。门面服务组装响应。

### 汇总页

`MerchantReconciliationSummaryBuilder` 读取匹配后的明细聚合结果，再合并红单费用、期初余额和银行流水金额。`MerchantBankFlowMatcher` 负责银行流水桥接红单付款及分摊。

### 导出

明细导出和汇总导出复用页面同一套 matcher 和 builder 输出。导出器只负责工作簿写入和格式化，不参与业务计算。

## 错误处理

- 导入器对用户可修复问题抛出业务可读的 `ValueError`，例如 sheet 缺失、月份不符、店铺无法唯一识别、结算状态非法。
- 门面服务负责把业务错误转换为现有 API 响应格式。
- 匹配失败返回业务状态，不抛异常。
- 汇总构建遇到缺失期初余额、无银行流水、无红单上下文时生成空值或 `pending` 状态。
- 只有账期非法、权限非法、文件无法读取这类输入问题才提前失败。

## 中文注释规范

重构代码应补充中文注释，但只解释非显然业务规则：

- 为什么货款 sheet 需要通过采购 sheet 推断店铺。
- 为什么红单匹配使用商品编码，再使用商家、直播日期、直播间匹配货款。
- 为什么匹配失败是业务结果，不是系统异常。
- 为什么银行流水优先桥接红单付款。
- 为什么银行流水需要按汇总权重分摊。
- 为什么兜底主体匹配只作为兼容路径。

不应为简单字段赋值、普通循环、显而易见的查询条件添加注释，避免注释噪音。

## 测试策略

### 回归基线

保留并继续运行现有 `backend/app/tests/test_merchant_reconciliation_service.py`。重构完成后，现有测试应不改业务断言即可通过。

### 模块级测试

新增或拆分测试覆盖：

- 红单模板和红单 sheet 解析。
- 货款结算状态校验。
- 货款店铺推断。
- 银行流水文件名解析。
- 银行流水格式识别和直播日期解析。
- 商品编码匹配红单采购。
- 采购行匹配货款行。
- 银行流水桥接红单付款。
- 银行流水按汇总权重分摊。
- 汇总金额刷新和银行状态计算。

### 等价性测试

使用同一批 mock 动账、红单、银行流水和期初余额数据，断言重构后的明细 payload、汇总金额、银行流水金额和银行状态与当前服务输出一致。

## 后续预留

第一阶段完成后，可以在不改 API 的前提下继续演进：

- 增加标准化 `live_date_key`，降低日期文本匹配风险。
- 增加主体别名表，降低公司名称弱匹配风险。
- 增加商家对账明细结果快照表和汇总快照表。
- 将银行流水未匹配原因结构化，支持人工修正。
- 给汇总页增加匹配方式标记，区分红单桥接和主体兜底。

## 验收标准

- 公共 API 行为和返回字段保持兼容。
- 现有商家对账测试通过。
- 新增模块级测试覆盖主要业务规则。
- `MerchantReconciliationService` 不再承载全部实现细节，而是作为门面协调模块。
- 关键业务规则有清晰中文注释，且没有低价值注释噪音。
