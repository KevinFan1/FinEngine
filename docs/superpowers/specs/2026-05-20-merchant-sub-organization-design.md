# 商家子组织设计

## 背景

当前系统用 `org_id` 作为上传、任务、店铺、汇总、动账核算、审计和配额的主要数据隔离字段。新需求是组织管理员可以为总公司增加商家，每个商家本质上是一个独立组织。商家不能看到总公司的数据，总公司能看到商家的数据。

本设计把“商家”建模为组织的直属子组织，而不是把商家压进现有店铺表。这样可以复用当前基于 `org_id` 的隔离模型，并让商家拥有自己的用户、店铺、上传和汇总数据。

## 目标

- 总公司组织管理员可以创建和管理直属商家。
- 商家是独立组织，有自己的用户、店铺、上传、任务、汇总和动账核算数据。
- 商家管理员可以维护自己组织的用户和店铺。
- 商家管理员不能创建子公司，不能管理总公司或其他商家。
- 总公司组织管理员可以查看总公司和直属商家的业务数据。
- 商家用户只能查看自己组织的数据。
- 配额容量单位在前端和接口展示中统一为 MB。
- 默认账号限制改为 3 个。
- 组织管理员创建商家时，商家固定为 3 个账号和 300 MB 月上传额度，组织管理员不能修改。
- 超级管理员创建或维护组织、商家和配额时不受上述固定额度限制。

## 非目标

- 不支持多级商家结构。商家下面不能再创建商家。
- 不引入独立的“商户账号体系”。用户仍然绑定 `fin_users.org_id`。
- 不改变现有上传文件、任务、汇总等核心表的 `org_id` 归属方式。
- 不把店铺等同于商家。店铺仍然属于某个组织，商家组织下面可以有多个店铺。

## 数据模型

### `fin_organizations`

新增字段：

- `parent_id bigint null`：父组织 ID。总公司为空，商家指向总公司。
- `org_type varchar(20) not null default 'company'`：组织类型。
  - `company`：总公司或普通组织。
  - `merchant`：商家子组织。

建议索引：

- `idx_fin_org_parent_id(parent_id)`
- `idx_fin_org_parent_type(parent_id, org_type)`

约束规则：

- `merchant.parent_id` 必须存在。
- `company.parent_id` 通常为空。
- 不允许 `merchant` 作为父组织再创建商家。
- 组织名称和组织编码继续保持全局唯一，避免跨总公司同名造成管理歧义。

### 配额字段

继续使用现有组织配额字段：

- `max_users`
- `max_storage_bytes`

业务语义调整：

- `max_storage_bytes` 仍然以字节存储。
- API 和前端展示单位改成 MB。
- 本字段继续表示“每月上传额度”，避免再显示成存储容量。

默认值调整：

- 新建普通组织默认 `max_users = 3`。
- 新建普通组织默认月上传额度可由超管设置；如果不设置，后端默认值按产品默认策略给出。
- 组织管理员创建商家时固定：
  - `max_users = 3`
  - `max_storage_bytes = 300 * 1024 * 1024`
- 超级管理员创建或编辑组织、商家时可以修改 `max_users` 和月上传额度。

## 权限模型

权限拆成两类：数据可见范围和管理动作范围。

### 角色

沿用现有角色：

- `superadmin`
- `org_admin`
- `member`

不新增 `merchant_admin` 角色。商家管理员仍然是 `org_admin`，能力由所属组织类型和父子关系决定。

### 数据可见范围

| 用户类型 | 可见组织范围 |
| --- | --- |
| `superadmin` | 全部组织 |
| 总公司 `org_admin` | 自己组织 + 直属商家组织 |
| 商家 `org_admin` | 自己组织 |
| `member` | 自己组织 |

后端新增统一权限服务，避免各接口继续散落判断：

- `get_visible_org_ids(user)`：返回当前用户可查看的组织 ID 集合。
- `assert_org_visible(user, org_id)`：目标组织不可见时返回业务错误。
- `resolve_query_org_ids(user, requested_org_id)`：列表查询时合并用户权限和前端筛选。

### 管理动作范围

| 动作 | `superadmin` | 总公司 `org_admin` | 商家 `org_admin` | `member` |
| --- | --- | --- | --- | --- |
| 创建总公司组织 | 可以 | 不可以 | 不可以 | 不可以 |
| 创建商家 | 可以 | 可以创建直属商家 | 不可以 | 不可以 |
| 编辑商家基础信息 | 可以 | 只能编辑直属商家 | 只能编辑自己组织的非配额信息 | 不可以 |
| 修改商家账号和容量额度 | 可以 | 不可以 | 不可以 | 不可以 |
| 停用商家 | 可以 | 只能停用直属商家 | 不可以 | 不可以 |
| 维护用户 | 可以 | 自己组织 + 直属商家 | 自己组织 | 不可以 |
| 维护店铺 | 可以 | 自己组织 + 直属商家 | 自己组织 | 不可以 |

新增统一权限判断：

- `can_create_merchant(operator, parent_org_id)`
- `can_manage_org(operator, target_org_id)`
- `can_manage_users(operator, target_org_id)`
- `can_manage_shops(operator, target_org_id)`
- `can_update_quota(operator, target_org_id)`

关键规则：

- 商家组织不能作为 `parent_id` 创建新商家。
- 组织管理员创建商家时，后端忽略前端传入的配额字段，强制写入 3 个账号和 300 MB。
- 组织管理员不能修改任何组织的配额。
- 超级管理员不受固定额度限制。

## 后端接口设计

### 组织接口

保留现有 `/organizations`，新增商家相关能力。

建议新增：

- `GET /organizations`
  - `superadmin` 返回全部。
  - 总公司 `org_admin` 返回自己和直属商家。
  - 商家 `org_admin` 返回自己。
  - 支持 `org_type` 和 `parent_id` 筛选。

- `POST /organizations`
  - `superadmin` 可以创建 `company` 或 `merchant`。
  - 非超管不用于创建普通公司。

- `POST /organizations/{parent_org_id}/merchants`
  - 总公司 `org_admin` 创建直属商家。
  - 商家 `org_admin` 请求该接口返回业务错误。
  - 后端固定 `org_type = merchant`、`parent_id = parent_org_id`、`max_users = 3`、`max_storage_bytes = 300MB`。

- `PUT /organizations/{org_id}`
  - 超管可编辑全部字段和配额。
  - 总公司 `org_admin` 可编辑直属商家的基础信息，不能传或修改配额。
  - 商家 `org_admin` 可编辑自己组织的基础信息，不能修改 `parent_id`、`org_type`、配额。

- `PUT /organizations/{org_id}/status`
  - 超管可操作全部组织。
  - 总公司 `org_admin` 只能停用或启用直属商家。

### 用户接口

现有 `UserService` 中按 `operator.org_id` 判断的逻辑改为权限服务：

- 列表：使用 `resolve_query_org_ids`。
- 创建用户：目标 `org_id` 必须在 `can_manage_users` 范围内。
- 更新用户：目标用户所属组织必须在 `can_manage_users` 范围内。
- 商家自己的用户数量配额按商家组织单独计算。
- 总公司为商家创建用户时，也消耗商家自己的 3 个账号额度。

### 店铺接口

现有 `ShopService` 中按 `shop.org_id == operator.org_id` 的判断改为：

- 列表：用可见组织范围过滤；总公司管理员可以按商家筛选。
- 创建：目标 `org_id` 必须在 `can_manage_shops` 范围内。
- 更新和删除：店铺所属组织必须在 `can_manage_shops` 范围内。

前端创建店铺时：

- 总公司管理员需要可以选择“总公司/商家”作为店铺所属组织。
- 商家管理员不显示组织选择，默认自己的组织。

### 业务数据接口

以下查询需要从单一 `org_id == current_user.org_id` 改成可见组织范围：

- 上传批次和文件列表。
- 任务列表、任务详情、重跑和进度接口。
- 汇总明细、汇总报表、导出接口。
- 汇总调整。
- 店铺筛选项。
- 审计日志。
- 动账核算科目、规则、上传、任务、明细、汇总和导出。

数据写入仍然只写入一个明确 `org_id`：

- 商家上传的数据归属商家组织。
- 总公司替商家上传时，前端必须明确选择目标组织；后端校验该目标组织在可管理范围内。
- 总公司上传自己的数据时，目标组织为总公司。

### 配额接口

接口语义统一为月上传额度：

- `GET /quota/me`
  - 返回当前用户所属组织自己的配额。
  - 商家看到商家自己的 3 账号、300 MB。

- `GET /quota/{org_id}`
  - 超管可查看任意组织。
  - 总公司 `org_admin` 可查看自己和直属商家。
  - 商家 `org_admin` 只能查看自己。

- `POST /quota/{org_id}/update`
  - 仅超管可修改。
  - 单位支持 MB 输入，后端转字节存储。

返回字段建议新增或调整：

- `users.current`
- `users.max`
- `monthly_upload.current_mb`
- `monthly_upload.max_mb`
- `monthly_upload.usage_percent`

保留字节字段可用于兼容，但前端默认展示 MB。

## 前端设计

### 菜单

总公司组织管理员显示：

- 商家管理
- 用户管理
- 店铺管理
- 上传中心
- 任务列表
- 汇总明细/汇总报表
- 动账资金核算

商家组织管理员显示：

- 用户管理
- 店铺管理
- 上传中心
- 任务列表
- 汇总明细/汇总报表
- 动账资金核算

商家组织管理员不显示“商家管理”。

### 商家管理页

建议从现有组织管理页拆一个更聚焦的页面：

- 表格字段：商家名称、编码、状态、账号用量、月上传额度用量、创建时间。
- 操作：查看、编辑、启用/停用。
- 新增商家抽屉：
  - 商家名称。
  - 商家编码。
  - 备注。
  - 固定提示：账号上限 3 个，月上传额度 300 MB。
  - 不显示可编辑配额输入。

超管在组织管理页可继续编辑配额，单位从 GB 改成 MB。

### 用户管理页

总公司管理员：

- 组织筛选下拉包含自己和直属商家。
- 新增用户时可选择自己或直属商家。
- 角色不能选择 `superadmin`。

商家管理员：

- 不显示组织选择。
- 新用户默认绑定自己的商家组织。

### 店铺管理页

总公司管理员：

- 可按组织筛选。
- 新增店铺时可选择自己或直属商家。

商家管理员：

- 不显示组织选择。
- 新店铺默认绑定自己的商家组织。

### 配额展示

- 全部容量展示从 GB 改成 MB。
- 文案统一为“每月上传额度”。
- Header 里的额度组件展示当前用户所属组织自己的额度，不展示集团汇总额度。
- 商家列表里可以展示每个商家的账号和月上传额度使用情况。

## 迁移策略

1. 给 `fin_organizations` 增加 `parent_id` 和 `org_type`。
2. 现有组织默认迁移为 `org_type = company`、`parent_id = null`。
3. 现有组织默认账号限制调整为 3 时需要谨慎：
   - 对历史组织不强制降低已配置额度，避免已有组织突然无法继续创建用户。
   - 新建组织默认 3 个账号。
   - 如果需要统一历史免费组织为 3 个账号，应单独跑数据修正脚本并先确认影响。
4. 配额单位只是展示和接口输入调整，数据库继续存字节。

## 测试重点

后端测试：

- 总公司管理员可以创建直属商家，商家固定 3 账号和 300 MB。
- 商家管理员创建商家失败。
- 总公司管理员不能修改商家配额。
- 超管可以修改任意组织和商家配额。
- 商家管理员只能看到自己组织的数据。
- 总公司管理员能看到自己和直属商家的任务、汇总、店铺、用户和审计。
- 总公司管理员不能访问非直属商家。
- 商家用户数量达到 3 后不能继续创建。
- 月上传额度超过 300 MB 时拦截上传。

前端测试：

- 总公司管理员可见“商家管理”。
- 商家管理员不可见“商家管理”。
- 组织管理员创建商家时没有可编辑配额输入。
- 超管组织配额编辑单位为 MB。
- 商家管理员用户和店铺页面不出现组织选择。
- 总公司管理员用户和店铺页面只能选择自己和直属商家。

## 实施顺序

1. 增加组织树字段和迁移。
2. 增加统一组织权限服务。
3. 改造组织和商家接口。
4. 改造用户、店铺管理权限。
5. 改造业务数据查询的可见组织范围。
6. 改造配额接口和 MB 展示。
7. 增加商家管理前端页面。
8. 调整用户、店铺、上传、汇总、任务和动账核算页面的组织筛选。
9. 补齐后端权限测试和前端构建验证。
