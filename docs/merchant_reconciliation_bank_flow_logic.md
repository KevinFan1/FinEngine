# 商家对账银行流水逻辑说明

## 目标

商家对账在原有动账明细、红单采购/付款、对账汇总的基础上，新增银行流水接入，用于校验红单付款是否真实从银行账户支出。

当前实现的判断链路是：

1. 通过银行流水文件名解析账期。
2. 只保留备注/用途/附言里包含“直播”的银行流水行。
3. 通过银行流水内容解析账户名称、对方户名、交易金额、备注/用途里的直播日期。
4. 将银行流水先匹配到红单付款行。
5. 如果同一红单付款对应多条汇总明细，则按汇总明细金额权重分摊银行流水金额。
6. 在商家对账汇总里展示银行流水金额、银行付款差异和银行流水状态。

## 文件命名规则

银行流水文件名必须包含账期和“银行流水”：

- `YYYYMM_银行流水_xxxx.xls`
- `YYYY-MM_银行流水_xxxx.xlsx`
- `YYYY年MM月_银行流水_xxxx.xls`

后端会从文件名解析 `accounting_year` 和 `accounting_month`，不会依赖前端传入账期作为最终依据。账期用于导入存储、覆盖同账户旧流水，以及解析直播日期跨年场景。

## 支持的银行流水格式

### 企业网银格式

识别表头需要包含：

- `账户名称`
- `交易时间`
- `对方户名`
- `备注`

主要字段来源：

- 账户名称：流水行里的 `账户名称`
- 对方户名：`对方户名`
- 交易时间：`交易时间`
- 支出金额：`借方发生额（支取）`
- 收入金额：`贷方发生额（收入）`
- 直播日期：从 `用途 / 备注 / 附言` 合并文本里解析
- 流水号：优先 `账户明细编号-交易流水号`

### 中国邮政格式

识别表头需要包含：

- `交易日期`
- `交易时间`
- `对方户名`
- `用途`

主要字段来源：

- 账户名称：优先从表头上方“户名”附近读取；样例里可以读到合并区域的 `F2`
- 对方户名：`对方户名`
- 交易日期/时间：`交易日期` + `交易时间`
- 支出金额：`支出金额`
- 收入金额：`收入金额`
- 直播日期：从 `用途 / 备注 / 附言` 合并文本里解析
- 流水号：优先 `交易流水号`，其次 `全局路由号`

## 金额口径

银行流水行会同时保存借方、贷方和净流出金额：

```text
flow_amount = 支出金额 - 收入金额
```

商家对账汇总只纳入 `flow_amount > 0` 的流水，也就是实际付款流出。收入、退款、冲正类流水会保留在银行流水明细页，但不会进入“银行流水汇总”金额。

银行流水导入前还会做业务文本过滤：`用途 / 备注 / 附言` 合并文本里必须包含“直播”。不含“直播”的转账、手续费、普通货款等流水不会导入到商家对账银行流水明细。

## 直播日期解析

当前支持两类日期表达：

- 单日：`2025-11-11`、`11月11日`
- 月日区间：`11月11-20日`、`11月11日-20日`

日期会标准化为红单同口径：

```text
2025-11-11 => 11月11日
11月11-20日 => 11月11-20日
```

银行流水明细页和匹配 key 都使用这个红单口径，不再展示 `2025-11-11~2025-11-20` 这种 ISO 区间。

## 与商家对账汇总的匹配逻辑

银行流水不会直接按账户名和汇总行硬匹配，而是优先通过红单付款行做桥接。

### 第一步：银行流水匹配红单付款

匹配键：

```text
组织ID + 账户名称 + 对方户名 + 直播日期
```

对应关系：

- 银行流水 `账户名称` 对红单付款 `settlement_subject`
- 银行流水 `对方户名` 对红单付款 `receipt_merchant`
- 银行流水解析出的直播日期 对红单付款 `live_date`

账户名和户名会做基础标准化：去除空格、统一大小写、统一中英文括号。

### 第二步：按汇总明细权重分摊

如果银行流水匹配到了红单付款，并且该付款行已经关联到汇总明细，则按每个汇总组的 `live_amount` 权重分摊银行流水金额。

这个设计是合理的，因为它避免了一个付款流水对应多个商品编码、多个直播明细时，银行流水金额被重复计入多行汇总。

### 第三步：兜底匹配汇总行

如果银行流水没有匹配到红单付款，系统会退回到账户名/对方户名维度：

```text
组织名称 + 账户名称 + 对方户名
```

然后尝试合并到汇总行：

```text
组织名称 + 我方主体 + 商家收款主体
```

这个兜底只适合账户名和汇总行主体名称完全一致的场景；如果汇总行展示的是商家名，而不是结算主体名，兜底可能无法命中。

## 汇总字段含义

商家对账汇总新增/使用的银行流水字段：

- `bank_flow_amount`：银行流水付款汇总金额。
- `bank_payment_diff`：银行流水金额与红单已付金额差异。
- `bank_status`：银行流水状态。

状态规则：

```text
bank_flow_amount = 0                  => pending
bank_flow_amount = paid_flow_amount   => matched
其他                                  => diff
```

其中：

```text
bank_payment_diff = bank_flow_amount - paid_flow_amount
unpaid_flow_amount = payable_goods_balance - paid_flow_amount
```

## 当前逻辑合理性检查

整体逻辑是合理的，尤其是“银行流水先匹配红单付款，再按汇总组权重分摊”的路径，符合商家对账的业务结构：

- 银行流水是付款事实。
- 红单付款是应付/已付管理口径。
- 动账明细是 GMV 和应付货款来源。
- 汇总行可能比付款行更细，需要分摊而不是直接 join。

但当前逻辑成立依赖以下前提：

1. 红单付款的 `settlement_subject` 与银行流水 `账户名称` 一致。
2. 红单付款的 `receipt_merchant` 与银行流水 `对方户名` 一致。
3. 红单付款的 `live_date` 与银行流水解析后的直播日期格式一致。
4. 一个组织同一账期、同一账户上传的新流水可以覆盖旧流水。
5. 对账主要关注银行实际付款流出，不把收入/退款计入付款核验。

## 主要风险和建议

### 1. 直播日期格式是最大匹配风险

银行流水会把 `2025-11-11`、`11月11日`、`11月11-20日` 尽量标准化为红单口径，但红单付款里的 `live_date` 如果写成 `11/11-11/20` 或其他文本，银行流水仍可能无法先匹配红单付款。

建议：给红单付款和银行流水都新增统一的 `live_date_key` 或 `live_start_date/live_end_date`，导入时统一标准化，匹配时不再使用原始文本。

### 2. 主体名称需要别名能力

当前只做了基础文本标准化，没有处理公司简称、错别字、括号后缀、个体户后缀、历史户名等业务别名。

建议：增加“主体别名表”，至少支持：

- 银行账户名称别名
- 商家收款户名别名
- 红单结算主体别名

### 3. 店铺筛选下银行流水没有天然店铺维度

银行流水本身没有 `shop_id`。当前汇总按店铺筛选时，银行流水只能通过已匹配红单付款和汇总权重分摊到当前可见汇总行；如果进入兜底匹配，存在把组织级银行流水误合并到某个店铺视图的可能。

建议：店铺维度报表优先展示通过红单付款桥接成功的银行流水；兜底匹配可以只在全组织视图启用，或增加“匹配方式”提示。

### 4. 多笔付款和多笔流水的对应关系仍是弱匹配

当前匹配键不包含金额和交易日期，只按主体 + 对方户名 + 直播日期。若同一直播日期、同一收款户名存在多笔红单付款，系统会取最新付款行作为桥接对象。

建议：后续可以加入金额接近、交易日期区间、流水号备注等匹配评分，形成“确定匹配 / 疑似匹配 / 未匹配”。

### 5. 日期解析覆盖面有限

当前只解析单日和简单区间。如果备注/用途写成多个区间、缺少“日”、跨月区间、自然语言备注，可能解析不到直播日期。

建议：保留当前规则作为第一版，后续在银行流水明细页增加“未解析直播日期”筛选，按真实失败样本扩充规则。

## 结论

当前版本可以作为第一版上线验证：它能解析两类样例流水，能只保留直播相关流水，能把银行付款事实接入商家对账汇总，并且核心分摊逻辑避免了重复计数。

上线前建议重点确认红单付款的直播日期是否已经和银行流水解析结果同口径。如果不一致，应优先补齐统一日期 key，否则会出现银行流水明细已导入、但汇总页银行金额无法命中的情况。
